# Zombie Branch Hunter

GitHub Action, который раз в неделю сам находит заброшенные ветки и
"зависшие" pull request'ы в твоём репозитории — и вместо казённого "This PR
has been inactive for 30 days" оставляет живое, человеческое сообщение,
сгенерированное AI на основе того, что реально происходило в этой ветке.

Полностью бесплатно: GitHub API, GitHub Actions и Google Gemini API (бесплатный
тир) — без единого платного компонента, без своего сервера, без базы данных.

---

## Зачем это нужно

В любом репозитории, который живёт больше пары месяцев, рано или поздно
накапливаются ветки-призраки: кто-то начал фичу, отвлёкся, забыл. PR висит
без ревью. Никто не хочет быть тем, кто напишет "удали уже эту ветку" — и
поэтому её никто не трогает годами.

Стандартные stale-боты решают это шаблонным текстом вроде:

> This pull request has been automatically marked as stale because it has
> not had recent activity. It will be closed if no further activity occurs.

Такое сообщение раздражает и его хочется проигнорировать — оно очевидно
писалось не для конкретно этой ветки. Zombie Branch Hunter вместо этого
смотрит на название ветки, последние коммиты и изменённые файлы, и пишет
что-то более похожее на:

**До (типичный stale-бот):**
> This PR has been inactive for 34 days. Please respond within 7 days or it
> will be automatically closed.

**После (Zombie Branch Hunter):**
> Привет! Смотрю на ветку `feature/csv-export` — судя по коммитам, ты
> дорабатывал экспорт больших таблиц и в последний раз упирался в
> тайм-ауты стрима. Работа ещё актуальна, или можно смело закрывать? Если
> просто руки не дошли доделать — дай знать, помогу разобраться с тем, где
> застряло.

## Как это работает

Раз в неделю (по расписанию, которое ты настраиваешь через cron) Action:

1. Сканирует все ветки репозитория и все открытые PR
2. Находит те, где не было коммитов больше 30 дней (порог настраивается)
3. Берёт название ветки, последние 3 коммит-мессаджа и список изменённых файлов
4. Отправляет это в Gemini API (бесплатный тир) с просьбой предположить, что
   это была за задача, и написать короткое дружелюбное сообщение
5. Если у ветки есть открытый PR — постит сообщение как комментарий в PR
6. Если PR нет (просто забытая ветка) — добавляет её в сводный Issue
   "🧟 Zombie Branches Report", который создаётся/обновляется каждую неделю
7. Проверяет, не комментировал ли бот этот же PR за последние 7 дней, чтобы
   не спамить

Всё выполняется внутри самого GitHub Actions runner — никакого отдельного
сервера, к которому нужно было бы получать доступ или который нужно было бы
поддерживать.

## Установка за 3 шага

### Шаг 1 — получи бесплатный API-ключ Gemini

1. Зайди на [aistudio.google.com](https://aistudio.google.com/app/apikey)
2. Войди через Google-аккаунт
3. Нажми **Get API key** → **Create API key**
4. Скопируй ключ 

Карту привязывать не нужно — бесплатный тир даёт достаточный дневной лимит
запросов для еженедельного скана ~5-15 веток.

### Шаг 2 — добавь workflow-файл и секрет в свой репозиторий

1. В своём репозитории создай файл `.github/workflows/weekly-scan.yml` со
   следующим содержимым (или скопируй файл
   [`.github/workflows/weekly-scan.yml`](.github/workflows/weekly-scan.yml)
   из этого репозитория):

   ```yaml
   name: 🧟 Zombie Branch Hunter — еженедельный скан

   on:
     schedule:
       - cron: '0 9 * * 1'  # каждый понедельник в 09:00 UTC
     workflow_dispatch: {}

   permissions:
     contents: read
     pull-requests: write
     issues: write

   jobs:
     hunt-zombies:
       runs-on: ubuntu-latest
       steps:
         - uses: actions/checkout@v4
         - uses: your-username/zombie-branch-hunter@v1
           with:
             gemini-api-key: ${{ secrets.GEMINI_API_KEY }}
             inactive-days-threshold: '30'
   ```

   Замени `your-username/zombie-branch-hunter@v1` на реальный путь к своему
   форку этого репозитория (см. блок "Форк vs копирование кода" ниже).

2. Добавь ключ из шага 1 в GitHub Secrets:
   - Зайди в свой репозиторий → **Settings** → **Secrets and variables** →
     **Actions**
   - Нажми **New repository secret**
   - Имя: `GEMINI_API_KEY`
   - Значение: ключ, который скопировал на шаге 1
   - Сохрани

### Шаг 3 — готово

Action будет запускаться автоматически каждый понедельник. Если хочешь
проверить его сразу, не дожидаясь понедельника — зайди во вкладку **Actions**
своего репозитория, выбери workflow "🧟 Zombie Branch Hunter", нажми **Run
workflow** и запусти вручную (с `dry_run: true` для первого раза — см. раздел
"Тестирование" ниже).

## Форк vs копирование кода

Есть два способа использовать этот Action:

**Вариант A — сослаться на этот репозиторий напрямую** (проще). В строке
`uses:` своего workflow-файла укажи `owner/zombie-branch-hunter@v1`, где
`owner` — тот, кто выложил этот код на GitHub (например, твой форк). Для
этого в исходном репозитории должен существовать git-тег/релиз `v1` —
создать его можно командой:

```bash
git tag v1
git push origin v1
```

**Вариант B — скопировать файлы прямо в свой репозиторий.** Скопируй
`action.yml` и папку `src/` в поддиректорию своего репозитория (например,
`.github/actions/zombie-branch-hunter/`), и в workflow-файле укажи
`uses: ./.github/actions/zombie-branch-hunter`. Так Action не будет зависеть
от внешнего репозитория вообще.

Для MVP на выходные вариант A обычно быстрее — не нужно ничего копировать
между репозиториями.

## Тестирование локально перед первым реальным запуском

Прежде чем пускать бота на настоящий рабочий репозиторий, стоит проверить
его на "сухую" — либо локально на своём компьютере, либо через
`workflow_dispatch` с `dry_run: true`.

### Вариант 1 — локально на компьютере (нужен установленный Node.js 20+)

1. Склонируй репозиторий и установи зависимости:

   ```bash
   git clone https://github.com/your-username/zombie-branch-hunter.git
   cd zombie-branch-hunter
   npm install
   ```

2. Скопируй пример конфига и заполни его:

   ```bash
   cp .env.example .env
   ```

   Открой `.env` в любом текстовом редакторе и заполни:
   - `GITHUB_TOKEN` — Personal Access Token с правами на чтение веток/PR и
     запись комментариев/issues (создать: GitHub → **Settings** →
     **Developer settings** → **Personal access tokens** → **Fine-grained
     tokens** → выбери репозиторий, права `Contents: Read`,
     `Pull requests: Read and write`, `Issues: Read and write`)
   - `GITHUB_REPOSITORY` — репозиторий, на котором тестируешь, в формате
     `owner/repo` (лучше взять тестовый/некритичный репозиторий, а не
     рабочий продакшен)
   - `GEMINI_API_KEY` — ключ из Шага 1
   - `DRY_RUN=true` — **обязательно оставь `true` для первого запуска**,
     чтобы Action ничего реально не постил, а только показал в консоли, что
     бы он запостил

3. Запусти:

   ```bash
   npm start
   ```

   В консоли увидишь список найденных зомби-веток и то, какие комментарии
   бот бы оставил — но реально в GitHub ничего не изменится, пока
   `DRY_RUN=true`.

4. Если результат выглядит адекватно — поменяй в `.env` `DRY_RUN=false` и
   запусти ещё раз, уже на тестовом репозитории, чтобы увидеть, как
   комментарии выглядят по-настоящему.

### Вариант 2 — через workflow_dispatch с dry-run прямо в GitHub

Если не хочешь ничего ставить локально, можно проверить прямо в GitHub:

1. Убедись, что в твоём workflow-файле есть `workflow_dispatch` с полем
   `dry_run` (пример есть в
   [`.github/workflows/weekly-scan.yml`](.github/workflows/weekly-scan.yml)
   этого репозитория)
2. Во вкладке **Actions** своего репозитория выбери workflow, нажми **Run
   workflow**, поставь `dry_run: true`, запусти
3. Открой лог запущенного job'а — там будет видно, какие ветки нашлись и
   какие сообщения бот бы написал, без реального постинга
4. Когда всё устраивает — запусти ещё раз с `dry_run: false`, либо просто
   дождись автоматического запуска по расписанию

## Настройки (inputs)

Все настраиваются в блоке `with:` workflow-файла:

| Параметр | По умолчанию | Описание |
|---|---|---|
| `gemini-api-key` | — (обязательный) | Ключ Gemini API, бери из Secrets, не пиши напрямую |
| `github-token` | `${{ github.token }}` | Обычно трогать не нужно — встроенный токен уже подходит |
| `gemini-model` | автоопределение | Явное имя модели, например `gemini-2.0-flash` |
| `inactive-days-threshold` | `30` | Через сколько дней без коммитов ветка считается заброшенной |
| `dedup-window-days` | `7` | Не постить повторно в тот же PR чаще этого срока |
| `gemini-request-delay-ms` | `4500` | Пауза между вызовами Gemini API (защита от лимита) |
| `gemini-max-retries` | `5` | Сколько раз повторять запрос при ошибке rate limit |
| `dry-run` | `false` | Если `true` — ничего не постит, только логирует |

## Ограничения бесплатного тира, о которых стоит знать

- **Gemini API (бесплатный тир):** лимит запросов в минуту и в день зависит
  от модели и может меняться — Action уже сделан с паузой между запросами
  (`gemini-request-delay-ms`) и retry с exponential backoff, так что при
  разумном количестве веток (до нескольких десятков за прогон) в лимит
  упереться сложно. Если всё же упрётесь — Action пропустит проблемную
  ветку, залогирует ошибку и продолжит с остальными, а не упадёт целиком.
- **GitHub Actions:** для публичных репозиториев минуты бесплатны без
  ограничений; для приватных — щедрый бесплатный лимит в месяц, которого
  для одного запуска в неделю с запасом хватит.
- **GitHub API:** встроенный `GITHUB_TOKEN` имеет свой лимит запросов в час,
  которого более чем достаточно для сканирования одного репозитория раз в
  неделю.

## FAQ

**Бот случайно закроет мою ветку/PR?**
Нет. Action только оставляет комментарии и обновляет сводный Issue — он
никогда не закрывает и не удаляет ветки или PR сам.

**Что если Gemini не ответит вовремя (rate limit)?**
Action сделает до 5 повторных попыток с увеличивающейся паузой между ними.
Если и это не поможет — конкретная ветка будет пропущена в этом прогоне (с
пометкой в логе), но остальные ветки всё равно обработаются, и через неделю
эта же ветка попробуется снова.

**Можно ли запускать чаще, чем раз в неделю?**
Да, поменяй cron-выражение в `weekly-scan.yml`. Но учти лимиты бесплатного
тира Gemini при большом количестве веток.

**Работает ли это с приватными репозиториями?**
Да, встроенный `GITHUB_TOKEN` имеет доступ к репозиторию, в котором запущен
workflow, независимо от того, публичный он или приватный.
